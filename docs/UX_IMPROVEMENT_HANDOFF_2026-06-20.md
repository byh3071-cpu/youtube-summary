# Focus Feed UX 개선 핸드오프

## 목적

2026-06-20 실사용자 관점 브라우저 점검에서 확인한 웹/모바일 UX 문제를 작은 구현 단위로 나눈다. 이 문서는 코드 변경 문서가 아니라 다음 구현 세션이 바로 작업을 시작할 수 있게 만든 Goal/Phase/Task 스펙이다.

## 점검 기준

- 데스크톱: 1440x900 Chromium 기준.
- 모바일: Pixel 5 크기 Chromium 기준.
- dev 서버에서는 HMR WebSocket 오류와 함께 일부 클릭 핸들러가 살아나지 않는 현상이 있었다. 실제 UX 판단은 production `next start` 기준으로 했다.
- 주요 관찰 흐름: 홈 첫 화면, 모바일 메뉴, 필터 열기/접기, 피드 Q&A, My Focus, 유튜브 카드, 라디오 추가, 하단 라디오 플레이어.

## 제품 방향

Focus Feed의 강점은 단순 피드 리더가 아니라 "AI가 내 관심사 기준으로 볼 것과 들을 것을 골라주는 개인 피드 DJ"라는 점이다. UX 개선은 기능을 늘리기보다 아래 세 가지를 먼저 선명하게 만드는 방향으로 진행한다.

1. 피드 카드에서 핵심 행동을 한눈에 고르게 한다.
2. 모바일 첫 화면에서 실제 콘텐츠까지 빠르게 도달하게 한다.
3. Q&A, 라디오, 요약, 북마크가 서로 겹치지 않는 안정적인 플로팅 레이어 체계를 만든다.

## 완료 목표

이 핸드오프의 완료 정의:

- 모바일/데스크톱 유튜브 카드 액션 버튼이 세로로 쪼개지지 않는다.
- AI 요약 버튼 이미지가 깨지지 않는다.
- 모바일에서 Q&A, 맨 위로, 필터, 라디오 푸터가 서로 클릭을 가로막지 않는다.
- 모바일 첫 화면에서 환영 배너와 히어로가 콘텐츠 진입을 과도하게 밀어내지 않는다.
- 모바일 메뉴가 "로고 클릭"이 아니라 명확한 메뉴 진입점으로 인식된다.
- Playwright 또는 수동 QA로 위 항목이 재현/검증된다.

## 우선순위 표

| 우선순위 | 영역 | Goal | 완료 기준 |
|---|---|---|---|
| P0 | 시각적 파손 | 카드 액션/아이콘 깨짐 제거 | 버튼 텍스트 줄바꿈 없음, 이미지 400 없음 |
| P0 | 클릭 가능성 | 모바일 플로팅 레이어 충돌 제거 | Q&A/맨 위로/라디오/필터가 모든 상태에서 클릭 가능 |
| P1 | 모바일 첫 화면 | 첫 진입 밀도 개선 | 첫 화면에서 My Focus 또는 피드 진입점이 자연스럽게 보임 |
| P1 | 발견성 | 모바일 메뉴 affordance 개선 | 메뉴 버튼임을 시각적으로 알 수 있음 |
| P2 | 제품 차별화 | 개인 피드 DJ 흐름 구체화 | 후속 PRD/스펙에 큐레이션 UX가 정의됨 |

  - 필터 확장 상태에서 Q&A/맨 위로/라디오 푸터 클릭 충돌 여부가 확인된다.

  - 구현자가 코드 수정 전/후 같은 체크리스트로 비교할 수 있다.

## Phase 1 — P0 시각적 파손 제거

목표: 사용자가 즉시 "깨졌다"고 느낄 수 있는 요소를 먼저 고친다.

### UX-10. AI 요약 아이콘 경로 정리

- 관찰:
  - `/images/icons/AI_summary.png` 요청이 400으로 실패한다.
  - 실제 파일은 `public/images/icons/AI_summary1.png` 등으로 존재한다.
- 파일 후보:
  - `src/components/feed/SummarizeButton.tsx`
  - `src/components/feed/InsightButton.tsx`
  - `src/components/ui/ThemeIcon.tsx`
  - `public/images/icons/`
- 구현 스펙:
  - `ThemeIcon name="AI_summary"`가 실제 존재하는 파일을 가리키게 한다.
  - 라이트/다크 아이콘 명명 규칙을 하나로 맞춘다.
  - 없는 아이콘 요청 시 깨진 이미지 대신 안전한 fallback 또는 lucide 아이콘을 보여준다.
- 완료 기준:
  - 브라우저 네트워크 탭에 AI 요약 아이콘 400이 없다.
  - 요약/인사이트 버튼 앞에 깨진 이미지 표시가 없다.
  - 다크 모드에서도 아이콘이 정상 표시된다.

### UX-11. 카드 액션 버튼 줄바꿈 금지

- 관찰:
  - `라디오에 추가`, `큐에 있음`, `딥다이브` 텍스트가 카드 안에서 세로로 쪼개진다.
  - 모바일 2열 카드에서는 특히 "라/디/오/에/추/가"처럼 보여 제품 완성도를 크게 해친다.
- 파일 후보:
  - `src/components/feed/AddToRadioButton.tsx`
  - `src/components/feed/YouTubeCard.tsx`
  - `src/components/feed/VideoDigestDrawer.tsx`의 `DeepDiveButton`
- 구현 스펙:
  - 액션 버튼에는 `whitespace-nowrap` 또는 아이콘 전용 모바일 변형을 적용한다.
  - 모바일 2열 카드에서는 긴 텍스트 버튼을 작은 카드 내부에 억지로 넣지 않는다.
  - 카드 액션 영역은 고정 높이를 유지해 버튼 상태 변화가 레이아웃을 흔들지 않게 한다.
- 완료 기준:
  - 데스크톱 4열 카드와 모바일 2열 카드에서 버튼 텍스트가 세로로 줄바꿈되지 않는다.
  - `큐에 있음` 상태로 바뀌어도 카드 높이가 과도하게 튀지 않는다.
  - 버튼 중심부가 다른 요소에 가려지지 않는다.

### UX-12. 카드 액션 우선순위 재정렬

- 관찰:
  - 카드마다 북마크, 라디오, 더보기, 요약, 딥다이브가 경쟁한다.
  - 지금은 기능이 많은 것이 장점보다 복잡함으로 보인다.
- 파일 후보:
  - `src/components/feed/YouTubeCard.tsx`
  - `src/components/feed/FeedItem.tsx`
- 구현 스펙:
  - 1차 액션: `요약`, `라디오`, `저장`.
  - 2차 액션: `딥다이브`, `더보기`, 외부 열기는 overflow 메뉴 또는 하단 보조 줄로 이동.
  - 모바일은 아이콘 중심, 데스크톱은 icon+short label 중심으로 분기한다.
- 완료 기준:
  - 사용자가 카드에서 해야 할 첫 행동 3개가 명확하다.
  - 카드 하단 액션 행이 두 줄 이상으로 어수선하게 늘어나지 않는다.

## Phase 2 — P0 모바일 플로팅 레이어 안정화

목표: 하단에 떠 있는 기능들이 서로 클릭을 가로막지 않게 한다.

### UX-20. 플로팅 스택 규칙 정의

- 관찰:
  - Q&A 버튼, 맨 위로 버튼, 라디오 빈 안내/플레이어, 필터 확장 상태가 같은 오른쪽 하단 영역을 차지한다.
- 파일 후보:
  - `src/components/feed/FeedQADrawer.tsx`
  - `src/components/ui/ScrollToTop.tsx`
  - `src/components/player/FloatingRadioPlayer.tsx`
  - `src/components/layout/AppLayout.tsx`
- 구현 스펙:
  - 모바일 floating layer 순서를 문서화하고 CSS 변수 또는 공통 상수로 정리한다.
  - 예: 하단 라디오 영역 위에 Q&A, Q&A 위/아래에 ScrollToTop 중 하나만 노출.
  - 필터 확장 중에는 ScrollToTop과 Q&A가 겹치지 않도록 위치를 올리거나 하나를 숨긴다.
- 완료 기준:
  - Q&A 버튼 중심 좌표에서 `elementFromPoint`가 Q&A 버튼 또는 그 자식이다.
  - ScrollToTop 버튼 중심 좌표에서 ScrollToTop이 클릭된다.
  - 라디오 플레이어가 열린 상태에서도 Q&A 드로어 입력/전송 버튼이 가려지지 않는다.

### UX-21. Q&A 버튼 상태별 클릭성 회귀 테스트

- 파일 후보:
  - `e2e/mobile-ux.spec.ts`
- 구현 스펙:
  - 초기 홈에서 Q&A 열기.
  - 필터 열기 후 Q&A 열기.
  - 라디오 큐 추가 후 Q&A 열기.
  - ScrollToTop 노출 상태에서 Q&A 열기.
- 완료 기준:
  - 모든 상태에서 `role=dialog` `피드 Q&A`가 열린다.
  - body scroll lock이 적용되고 Escape로 닫힌다.

### UX-22. 빈 라디오 안내 위치/해제 UX 정리

- 관찰:
  - 모바일 첫 화면에서 빈 라디오 안내가 콘텐츠를 가리고, 닫기 버튼도 다른 플로팅 버튼과 경쟁한다.
- 파일 후보:
  - `src/components/player/FloatingRadioPlayer.tsx`
- 구현 스펙:
  - 첫 방문 안내는 짧게 보이고 dismiss 이후 재노출하지 않는다.
  - 모바일에서는 카드/필터 조작 중 안내가 주요 버튼을 덮지 않게 한다.
- 완료 기준:
  - 빈 안내 닫기 후 같은 세션에서 재등장하지 않는다.
  - 안내가 Q&A/ScrollToTop 중심 클릭을 가로막지 않는다.

## Phase 3 — P1 모바일 첫 화면과 메뉴 발견성

목표: 모바일 첫 화면이 제품 설명보다 실제 사용 흐름을 먼저 보여주게 한다.

### UX-30. 모바일 히어로 높이 축소

- 관찰:
  - 모바일 첫 화면에서 큰 히어로 카드가 피드 진입을 밀어낸다.
- 파일 후보:
  - `src/components/feed/FeedHeader.tsx`
  - `src/app/page.tsx`
- 구현 스펙:
  - 모바일에서는 히어로 일러스트를 숨기거나 축소한다.
  - `총 n개`, `새로고침`, 현재 필터 상태만 남긴 compact header를 고려한다.
- 완료 기준:
  - 모바일 첫 viewport 안에서 My Focus 또는 필터/피드 섹션이 자연스럽게 보인다.
  - 데스크톱 히어로의 넓은 여백 감성은 유지된다.

### UX-31. 환영 배너 first-run 처리

- 관찰:
  - 환영 배너가 기능 소개는 잘하지만, 모바일에서 첫 콘텐츠 도달을 늦춘다.
- 파일 후보:
  - `src/components/feed/WelcomeBanner.tsx`
- 구현 스펙:
  - 닫기 상태를 localStorage에 저장한다.
  - 모바일에서는 3개 카드형 설명 대신 한 줄 요약 + "자세히" 또는 접힘 상태로 제공한다.
- 완료 기준:
  - 닫은 배너가 새로고침 후 다시 나타나지 않는다.
  - 모바일 첫 화면에서 배너가 화면 대부분을 차지하지 않는다.

### UX-32. 모바일 메뉴 버튼 명확화

- 관찰:
  - 현재 로고 전체가 `메뉴 열기` 버튼이다. 사용자는 로고로 인식할 가능성이 높다.
- 파일 후보:
  - `src/components/layout/MobileHeaderWithNav.tsx`
  - `src/components/layout/MobileNavDrawer.tsx`
- 구현 스펙:
  - 로고 옆 또는 왼쪽에 명확한 햄버거 아이콘/메뉴 텍스트를 둔다.
  - 로고 클릭은 홈 이동 또는 메뉴 열기 중 하나로 일관되게 정한다.
- 완료 기준:
  - 모바일 상단에서 메뉴 진입점이 시각적으로 명확하다.
  - 접근성 이름은 `메뉴 열기`를 유지한다.

## Phase 4 — P1 카드와 라디오 경험 정돈

목표: 라디오 모드를 Focus Feed의 대표 경험으로 끌어올린다.

### UX-40. 라디오 추가 후 피드백 강화

- 관찰:
  - 라디오 큐에 추가되면 하단 플레이어가 열리지만, 카드 쪽 변화가 세로 줄바꿈 때문에 어색하다.
- 파일 후보:
  - `src/components/feed/AddToRadioButton.tsx`
  - `src/components/player/FloatingRadioPlayer.tsx`
- 구현 스펙:
  - 카드 버튼은 `라디오` 또는 아이콘+짧은 상태로 축약한다.
  - 추가 직후에는 토스트 또는 하단 플레이어 상태로 "큐에 추가됨"을 명확히 알린다.
- 완료 기준:
  - 추가 전/후 카드 레이아웃이 안정적이다.
  - 사용자가 현재 몇 개가 큐에 있는지 알 수 있다.

### UX-41. 모바일 카드 밀도 조정

- 관찰:
  - 모바일 2열 카드는 썸네일과 제목은 좋지만 액션 행이 과밀하다.
- 파일 후보:
  - `src/components/feed/YouTubeCard.tsx`
  - `src/components/feed/FeedList.tsx`
- 구현 스펙:
  - 모바일 2열에서는 썸네일, 제목 2줄, 채널, 핵심 액션 2개 이하로 제한한다.
  - 나머지 기능은 카드 상세/더보기로 이동한다.
- 완료 기준:
  - 393px 폭에서 카드 액션이 2줄 이상 난잡하게 증가하지 않는다.
  - 제목과 썸네일이 주요 시각 정보로 남는다.

## Phase 5 — P2 제품 차별화 확장

목표: 단기 버그 수정 이후 Focus Feed만의 독창성을 제품 흐름으로 확장한다.

### UX-50. Today Queue / 피드 DJ 스펙

- 아이디어:
  - "10분만 듣기", "출근길 큐", "오늘 놓치면 안 되는 3개", "긴 영상만", "숏폼만" 같은 청취 큐를 AI가 만들어준다.
- 구현 전 스펙:
  - 사용자의 My Focus 목표를 입력으로 한다.
  - 피드 상위 항목에서 3~7개를 골라 큐를 제안한다.
  - 각 항목에 `왜 추천했는지`, `소요 시간`, `듣기/요약/저장` 액션을 붙인다.
- 완료 기준:
  - PRD 또는 별도 spec에 데이터 흐름과 UI 진입점이 정의된다.
  - 기존 AI 브리핑과 중복되지 않는 역할이 정리된다.

### UX-51. "왜 나에게 중요한가" 카드 배지

- 아이디어:
  - 카드에 단순 카테고리 대신 내 관심사와 연결되는 한 줄 이유를 표시한다.
- 구현 전 스펙:
  - 비로그인/목표 없음 상태에서는 기본 트렌드/카테고리 배지만 보여준다.
  - My Focus가 있으면 AI 브리핑 결과 또는 로컬 키워드 매칭으로 이유를 표시한다.
- 완료 기준:
  - 사용자가 "왜 이 콘텐츠가 나에게 뜨는지" 이해할 수 있다.

### UX-52. 저장 후 정리 흐름

- 아이디어:
  - 북마크/요약/노션/Todoist가 각각 흩어져 보이지 않도록 "나중에 정리" 흐름으로 묶는다.
- 구현 전 스펙:
  - 카드 저장 후 `북마크`, `요약`, `Todoist`, `Notion` 중 다음 액션을 제안한다.
  - 외부 서비스 연동은 실제 OAuth 전까지 빠른 링크/복사 기반으로 제한한다.
- 완료 기준:
  - 저장 후 사용자가 다음에 무엇을 할지 알 수 있다.

- 사용자 명시 없이 commit/push 하지 않는다.

- 기존 사용자 변경이 있는 파일은 되돌리지 않는다.
